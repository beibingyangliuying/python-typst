# Code Review

## 1. Overall Assessment

- The project is overall healthy: it has a clear core objective as a lightweight Typst code string generation library, a small source code footprint, a mirrored structure between `src/typstpy` and `tests/typstpy`, and test coverage for representative behaviors of the main public API.
- The biggest risk is not architectural drift but several local implementation flaws that directly affect generated output or public API behavior, especially the default naming in `attach_func`, the single-child spread rules for `pre_series`/`post_series`, and the use of `assert` for public validation.
- The protocol abstraction layer in `_core.py` is necessary and reasonable, preventing each Typst function from duplicating rendering logic; it should not be rewritten.
- The module boundaries in `std/layout.py`, `std/model.py`, `std/text.py`, and `std/visualize.py` are split along Typst standard library domains and are generally reasonable; no further splitting is needed for now.
- `visualize.py` is already nearing 1800 lines, but it mainly consists of homogeneous wrappers, so the benefit of splitting is limited. At this stage, fixing core protocol and testing gaps is more important than splitting for line count.
- `Document` is currently simple and usable, but the `@frozen` decorator conflicts with the internal mutable list builder behavior, misleading the API contract.
- Type annotations are a weakness for long-term evolution: many public functions lack return types and have broad parameter types. The project declares `>=3.10` but uses `typing.Self`, which is inconsistent with future type-checking goals. You have confirmed that the minimum Python version can be raised later.
- The engineering setup is lightweight with `pytest`, `ruff`, and `coverage` already configured, but it lacks a `build-system`, type-checking configuration, `py.typed`, and CI. Local reproducibility can still be strengthened.
- The highest priorities are: fixing confirmed correctness issues, adding corresponding regression tests, and then cleaning up exception types and type contracts.
- What can be kept as-is: the “thin function + _core protocol rendering” pattern for standard library wrappers, the mirrored test directory structure, and the main re-export approach in `__init__.py`.

## 3. Thematic Analysis

### Project Structure

- **What's good**: Mirror structure between `src/typstpy` and `tests/typstpy` is clear; `_core.py` holds protocol and rendering logic, `std/*` organized by Typst standard library domain, `customizations.py` for dynamic factories, `document.py` for document aggregation object; boundaries are generally reasonable.
- **Existing problems**: `visualize.py` is large but still homogeneous wrappers; `doc/code review/` is empty; root `readme.py` is a utility script but named like a regular module; `typstpy.egg-info`, `dist`, etc., exist in workspace but not tracked by git.
- **Modification suggested?**: Do not suggest large-scale splitting of modules; only suggest tidying `readme.py` script entry point and `tiling` export.
- **Modification risk**: Splitting `visualize.py` would affect import paths and README generation order; current benefit is insufficient.

### Abstraction Boundaries

- **What's good**: `normal`, `positional`, `instance`, `pre_series`, `post_series` in `_core` are necessary abstractions, preventing std wrappers from duplicating lots of string concatenation logic.
- **Existing problems**: `pre_series` / `post_series` simultaneously shoulder “rendering children order” and “single-argument spread” semantics; responsibilities slightly mixed; `attach_func` default naming bug shows decorator abstraction lacks direct tests.
- **Modification suggested?**: Suggest locally tightening protocol function contracts; do not suggest introducing class hierarchies or registry frameworks.
- **Modification risk**: `_core` is the foundation for all wrappers; any behavior change must include regression tests, especially for parameter omission and spread behavior.

### Implementation Quality

- **What's good**: Generation logic is simple and direct, mostly pure string transformations; most std functions just pass parameters; good locality.
- **Existing problems**: `assert`, `dict()` defaults, single-child spread, `import_`/`show_` special syntax, `__kwdefaults__ is None` are all low-level implementation flaws.
- **Modification suggested?**: Prioritize fixing small-scope issues in `_core`, then address std-layer value domain validation.
- **Modification risk**: Existing tests have solidified old outputs for `import_` and `show_(None)`; fixing will change observable behavior and requires a changelog or release note.

### Naming Consistency

- **What's good**: Python names basically follow Typst name mapping; `set_`, `show_`, `import_` avoid keyword conflicts; `hspace`/`vspace` are clearer than `h`/`v`.
- **Existing problems**: `bullet_list` corresponds to Typst `list`, `numbered_list` to `enum` – reasonable evasion but needs clear documentation; `numbering(numbering_)` parameter name slightly awkward but acceptable; `path`/`polygon` `fill_rule` value spelling inconsistent with `curve`; `attach_func` default naming behavior inconsistent with documented terminology.
- **Modification suggested?**: Do not suggest renaming public functions; only fix value spelling and documentation clarification.
- **Modification risk**: Changing public function names would break API, not recommended for now.

**Terms to unify:**

- Typst original function name: keep `original_name`.
- Generated Typst code: "Executable typst code" is acceptable, but suggest unifying capitalization as "Typst".
- Sequence parameter: keep `children`.
- Method call target: currently mixed `self`/`instance`; suggest internally unifying as `target` or keeping `self` with clear docstring.
- Fill rule: unify to `'"non-zero"'` and `'"even-odd"'`.

**Names recommended to keep:**

- `set_`, `show_`, `import_` – clear Python keyword conflict handling, already public API.
- `bullet_list`, `numbered_list` – more Pythonic than `list`, `enum`, avoid shadowing built-ins.
- `hspace`, `vspace` – more readable than `h`, `v`.
- `_core` – appropriate for expressing internal protocol layer.

**Names suggested to change or clarify:**

- `readme.py` – change to tool script semantics, like `scripts/generate_readme.py`, or at least add `main()`.
- `numbering_` – could keep; if changed to `pattern` or `numbering_pattern` would be clearer but is a public parameter name change, not recommended directly.
- `instance` parameter `instance` – internally could be renamed to `target`.

### Duplicate Logic

- **What's good**: Many std wrappers appear repetitive, but this is an explicit mapping of Typst API; keeping them is more readable than abstracting into a configuration table.
- **Existing problems**: Value domain validation sets scattered across functions; multiple relative validations repeated; `grid` and `table` wrappers are very similar.
- **Modification suggested?**: Do not suggest a config-driven rewrite to eliminate wrapper duplication; could extract a very small internal `_validate_choice`, but not first priority.
- **Modification risk**: Over-abstraction would make each Typst function’s parameter contract less intuitive.

### API & Data Contract

- **What's good**: Public API is mostly functional, returning strings, easy to test; `__all__` clarifies export boundaries.
- **Existing problems**: `Document` is a mutable builder marked `@frozen`; `image` bytes contract unclear; behavior of `.where`/`.with_` on functions without keyword-only defaults unclear; `import_` without `names` behavior unclear.
- **Modification suggested?**: Suggest making core contracts explicit but without changing the return type of most wrappers.
- **Modification risk**: Public wrappers all return strings; cannot easily change to an object model without breaking all callers.

### Type Annotations

- **What's good**: Already using `collections.abc`, `Self`, `ClassVar`, `overload`; direction is modern Python.
- **Existing problems**: Almost all public functions lack return types; most parameters lack types; `_render_value` `singledispatch` registers too broadly; no `py.typed`; no `mypy`/`pyright` config.
- **Modification suggested?**: Suggest adding types in phases: first give explicit return types to `_core` and `Document`, then gradually cover std wrappers.
- **Modification risk**: Adding all types at once causes lots of noise; suggest module-by-module progression.

### Modern Python Features

- **What's good**: Uses `positionnal-only`, `keyword-only`, `X | None`, `list[T]`, `attrs`, `Path`.
- **Existing problems**: `requires-python >=3.10` incompatible with `typing.Self`; `attrs` use in `Document` and `_Implement` yields limited benefit; mutable defaults don't follow modern practices.
- **Modification suggested?**: You've confirmed the version can be raised; suggest minimum `>=3.11` or, if project only uses 3.12, match `.python-version` to `>=3.12`.
- **Modification risk**: Raising minimum version is a packaging behavior change; should be documented in README/changelog.

### Functional Style & Side Effects

- **What's good**: Core wrappers are mostly pure functions, input to string output, easy to test.
- **Existing problems**: `implement` decorator attaches `.where`/`.with_` and writes to a global registry; `Document` is a mutable builder; `readme.py` has clipboard side effect.
- **Modification suggested?**: Keep wrapper pure function style; explicitly document decorator side effects; tidy `readme.py`.
- **Modification risk**: Removing all state is unrealistic; registry is the basis for README generation and Callable rendering.

### Return Value Design & Generators

- **What's good**: All generation functions return `str`, fitting callers' direct string concatenation consumption of Typst source.
- **Existing problems**: `Document.__str__` does not provide a `render()` explicit method, but `__str__` is sufficient for now; `_Implement.implement_table()` and `code_examples()` returning full strings is reasonable.
- **Modification suggested?**: Do not suggest introducing generators; returning `str` is the correct design.
- **Modification risk**: Changing return object or generator would break entire public API.

### Error Handling

- **What's good**: Many public functions already recognize the need for value domain validation.
- **Existing problems**: Use `assert`; error messages lack context; docstrings say `ValueError` but actual is `AssertionError`; `warnings.warn` doesn't specify warning class or stacklevel.
- **Modification suggested?**: Migrate to explicit exceptions per your confirmed direction.
- **Modification risk**: Tests currently expect `AssertionError`, must synchronize.

### Paths & I/O

- **What's good**: Core library has almost no file I/O; cross-platform risk is low.
- **Existing problems**: `image` source is path or bytes but only string rendering, does not handle `PathLike`; `readme.py` depends on clipboard.
- **Modification suggested?**: No need to introduce `pathlib` in core library; if supporting `Path`, just make conversion rule explicit.
- **Modification risk**: Automatically adding quotes around paths might change existing users' habit of handwriting Typst strings; not recommended for now.

### DataFrame Schema

- **What's good**: Project does not use pandas/DataFrame.
- **Existing problems**: None.
- **Modification suggested?**: Not needed.
- **Modification risk**: Not applicable.

### Determinism & Reproducibility

- **What's good**: Generation functions mainly determined by parameter order; Python 3.7+ dict order is stable.
- **Existing problems**: `_Implement.permanent` is a global registry; README table and examples output order depends on import/registration order; `WeakKeyDictionary` semantics also not suitable as long-term stable doc order source.
- **Modification suggested?**: Sort during README generation; do not change registry.
- **Modification risk**: Sorting will change README diff, but behavior becomes more reproducible.

### Performance

- **What's good**: Computation is small; `StringIO` used reasonably; no obvious performance bottleneck.
- **Existing problems**: `singledispatch` and recursive rendering may have overhead at massive nesting but not current bottleneck.
- **Modification suggested?**: Do not refactor for performance.
- **Modification risk**: Premature optimization reduces readability.

### Concurrency & Batch Execution

- **What's good**: Most wrappers are stateless pure functions; low concurrency calling risk.
- **Existing problems**: `_Implement.permanent` and `temporary` are global mutable registries; dynamic `customizations` register temporary functions.
- **Modification suggested?**: No need for concurrency isolation currently; if future needs batch README generation or multi-thread dynamic registration, then consider locks or independent registries.
- **Modification risk**: Registry refactoring would affect entire decorator mechanism, not recommended for now.

### Dependencies & Engineering Configuration

- **What's good**: Very few dependencies; runtime only `attrs` and `deprecated`; test tools centralized in `pyproject.toml`.
- **Existing problems**: No `[build-system]`; no type-checking configuration; no CI; `.python-version` inconsistent with `requires-python`; `format.ps1` only does import fix and format, not full lint.
- **Modification suggested?**: Suggest minimal supplementation, do not introduce heavy tools.
- **Modification risk**: Switching build backend may affect publishing; use `setuptools` minimal config or confirm current `uv build` dependency behavior.

### Testing

- **What's good**: Test structure is clear, covering public exports, core protocol, `Document`, customization factory, representative std functions.
- **Existing problems**: Tests lean towards happy path and few invalid params; lack coverage for `attach_func` default naming; missing single child series tests; missing `tiling` export test; missing correct Typst syntax tests for `import_` without names and `show_(None)`; tests depend on `AssertionError`.
- **Modification suggested?**: Must add critical regression tests.
- **Modification risk**: Correcting wrong behaviors requires updating existing test expectations; must be clear it's a bugfix.

### Documentation & Docstrings

- **What's good**: Most public functions have Google-style docstrings and examples; README coverage is broad.
- **Existing problems**: Docstrings have `_description_` placeholders, e.g., `subpar.py:35-57`; some `Raises` don't match actual exceptions; some default value restatements too long; README may be generated from unstable registry; `readme.py` has no entry protection.
- **Modification suggested?**: Synchronize error handling and public behavior first, then update README.
- **Modification risk**: README is a large file, easy to create noisy diffs; only generate after behavioral fixes.

### Backward Compatibility

- **What's good**: Already retained old Typst names via `deprecated`, e.g., `path`, `pattern`, `image.decode`.
- **Existing problems**: Fixing `import_`, `show_(None)`, single child spread may change existing output; raising Python minimum version affects install base.
- **Modification suggested?**: Correctness bugs can be fixed directly, but release note is needed.
- **Modification risk**: If users depend on old incorrect output, they'll be affected, but such dependence is low-value.

## 4. Evaluation of Alternative Implementation Approaches

| Location | Current Implementation | Main Issue | Alternatives |
|----------|------------------------|------------|--------------|
| `_core.attach_func` | Default name takes target function `func.__name__` | Default attach helpers missing or overwritten | A: Change to `attached.__name__`; B: Force explicit `name`; C: Introduce descriptor/namespace object |
| `_core.pre_series/post_series` | Single child always `..child` | Cannot express normal single child | A: String single child not spread, sequence single child spread; B: Add explicit `spread()` wrapper; C: Rewrite series protocol as structured AST |
| Public validation | Many `assert` | Validation disappears under `python -O` | A: Local `if` + `ValueError`; B: Small helper for unified validation; C: Full Enum/Literal modeling |
| `import_` / `show_` | Simple f-string, special cases not branched | Possible Typst syntax error | A: Special-case for no names / None element; B: New rule builder; C: AST-ify rules |
| `Document` | `@frozen` + internal list `append` | Misleading contract | A: Keep behavior, just change to plain mutable class; B: Immutable Document returning new object; C: dataclass slots builder |
| README generation | Registry order + clipboard | Non-deterministic, not automatable | A: Sort output and add `main`; B: Generate file command; C: Documentation system |

### Alternative A: Conservative Local Improvements

- **Core idea**: Do not refactor architecture, only fix confirmed `_core` bugs, Typst syntax edge cases, missing `__all__` export, and test gaps.
- **Files involved**: `_core.py`, `std/visualize.py`, `tests/typstpy/*`, `pyproject.toml`.
- **Scope of changes**: Small to medium, mainly local conditional branches and test expectations.
- **Pros**: Lowest risk, fastest elimination of P0/P1 correctness issues, preserves public API shape.
- **Cons**: Will not solve all type annotation and docstring quality issues at once.
- **Does it change public API?**: Minor bugfix-level behavioral changes, e.g., `import_(path)` and `show_(None)` output.
- **Need new tests?**: Must add regression tests.
- **Applicability**: Best fit for current stage, especially since you explicitly said "no large-scale refactoring first".

### Alternative B: Medium-scale Refactoring

- **Core idea**: On top of A, systematically migrate `assert` to explicit exceptions, tidy `Document` contract, add core type aliases and type-checking configuration.
- **Files involved**: `_core.py`, `document.py`, `std/*.py`, tests, `pyproject.toml`, README.
- **Scope of changes**: Medium, impacts many tests and docstrings.
- **Pros**: Significant improvement in long-term maintainability and type contracts.
- **Cons**: Broader change surface, easy to generate large mechanical diffs.
- **Does it change public API?**: Exception types change, Python minimum version changes; function names and return values preserved.
- **Need new tests?**: Need to update invalid parameter tests, add new behavior tests.
- **Applicability**: Second batch after P0/P1 are fixed.

### Alternative C: More Ideal but Riskier Design

- **Core idea**: Upgrade Typst code from string to lightweight AST / Code object, unified final render.
- **Files involved**: Almost all source and tests.
- **Scope of changes**: Large.
- **Pros**: Can thoroughly solve escaping, bytes, spread, syntax composition, type contracts and other long-term issues.
- **Cons**: High API breakage risk; existing README and tests would need full rewrite; current project scale does not justify the benefit.
- **Does it change public API?**: Most likely, unless many wrappers are retained.
- **Need new tests?**: Complete test rebuild required.
- **Applicability**: Not recommended for now; only consider when future needs complex Typst AST manipulation, escaping safety, and non-string composition.

**Recommended approach**: Start with Alternative A, then progressively absorb parts of Alternative B. The current biggest risks are a few local correctness issues, not architectural problems; A can reduce risk fastest without creating excessive abstraction.

## 5. Designs That Can Stay As-Is

- Callback-based architecture of `_core` and thin wrappers in `std` can stay. Each Typst function having a Python wrapper is repetitive but intuitive, not to be replaced by a big config table.
- `_core` protocol function abstraction is the core reuse point of the project, necessary and effective.
- Wrappers returning `str` fit the project positioning; callers consume easily.
- Underscore naming of `set_`, `show_`, `import_` balances Python keywords and Typst terms.
- `deprecated` functions keep retained for compatibility value, should not be deleted.
- No introduction of pandas/DataFrame schemas.
- No introduction of generators; returning strings suits debugging and concatenation better.
- Do not split `visualize.py`; file is large but homogeneous in responsibility, splitting adds import complexity.
- No introduction of complex type system to cover all Typst values; Typst expressions are open; over-narrowing reduces usability.

## 9. Final Conclusion

The core problem of the current project is not architectural complexity, but a few foundational protocol functions and public API contracts have local but wide-impact correctness risks. The most worth rethinking are the four categories of basic behaviors in `_core`: `attach`, series spread, rule rendering, and validation.  
The next step should first fix the P0/P1 issues in `_core` and protect behavior with small, clear regression tests. The large-area repetition in std wrappers can stay for now; don't rush to abstract into config tables or AST. `visualize.py` is big but not recommended to split yet. `Document`, type annotations, engineering configuration, and README generation can be gradually cleaned up as a second phase.  
Overall, the project is suitable for continued extension, but before extending, the underlying rendering protocol, exception contracts, Python version declaration, and test gaps should be stabilized; otherwise, future Typst API wrappers will keep replicating current problems.
