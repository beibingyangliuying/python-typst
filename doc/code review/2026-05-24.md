# Code Review: python-typst (v1.4.0-alpha.0)

## 1. Overall Assessment

1. **Clear code generator architecture**: The project uses three well-separated layers — `@implement` decorators to register Typst functions, an IR intermediate representation with a `singledispatch` rendering pipeline, and a `Document` builder — with clean responsibility boundaries.
2. **Highly consistent implementation pattern**: All functions under `std/` follow the same flow: parameter declaration → validation → protocol forwarding → rendering. This uniformity significantly reduces maintenance cost.
3. **Thorough verification system**: Every function includes a Typst official documentation verification comment (`# Typst docs verified on...`), parameter value validation, and doctest examples — rare among similar projects.
4. **Biggest risk**: The protocol functions in `_core/` (`normal`, `call_`, `instance`, etc.) are pure string concatenation engines without type-level safety guarantees; errors usually only surface when the generated Typst is compiled.
5. **Highest priority to fix**: Factory functions generated by `customizations.py` do not gain `where`/`with_` methods or parameter validation — this is a feature gap, not a design choice. The fragility of `strip_brace` (only removing the first and last characters) can produce incorrect output in edge cases.
6. **Designs that can stay as-is**: The `Document` builder is simple and practical enough; the docstring extraction and README generation in `_docs.py` works well; the constants in `_constants.py` are clearly defined.

## 3. Thematic Analysis

### 3.1 Project Structure & File Organization

- **Good**: Clear separation between `_core/` (internal engine) and `std/` (public API); re-exports in `__init__.py` have explicit `__all__`; test directory `tests/typstpy/` mirrors `src/typstpy/` well.
- **Problem**: `customizations.py` sits in the top-level package but is mainly used by advanced users; its architectural status is not entirely clear.
- **Suggestion**: The current location is acceptable; no need to move. If similar functionality grows in the future, an `ext/` or `contrib/` subpackage could be considered.

### 3.2 Abstraction Design & Responsibility Division

- **Good**: The combination of `@implement` decorator + `Implement` registry + `attach_func` is elegant; the protocol functions (`normal`, `instance`, `positional`, `pre_series`, `post_series`) abstract Typst's various calling patterns into five clear protocols.
- **Problem**: The `call_` protocol differs from `normal` only in argument order (`call_` doesn't handle body omission logic), essentially a variant of `normal`, increasing the number of protocols.
- **Suggestion**: Can stay as-is. Merging `call_` and `normal` would increase `normal`'s complexity with limited benefit.

### 3.3 Implementation Quality

- **Good**: The IR rendering design (`_to_expr` singledispatch → `RawExpr`/`SequenceExpr`/`MappingExpr` → `render()`) is clean, avoiding scattered `str()` and bracket concatenation; `_filter_default_kwargs` leveraging `__kwdefaults__` for default value filtering is a clever trick.
- **Problems**:
  1. `strip_brace(value: str) -> str: return value[1:-1]` is a fragile string operation (P0, see key issue table above).
  2. `_render_key` only does `str.replace('_', '-')`, not handling Typst's kebab-case naming convention (no impact for keys already containing `-`).
  3. Several protocol functions in `protocols.py` have repetitive `params` construction logic (create list, append, join).
- **Suggestions**:
  1. Fix `strip_brace`: replace with content-based parsing, or refactor IR nodes to provide an unwrapped rendering method that doesn't need `strip_brace`.
  2. Extract a `_render_params` helper to reduce duplication in `protocols.py`.

### 3.4 Naming Style & Terminology Consistency

- **Good**: Function names directly map to Typst function names (e.g., `align`, `grid`, `figure`); Python-specific functions use descriptive names (e.g., `bullet_list` for Typst `list`, `numbered_list` for `enum`); `_` prefix accurately marks private functions.
- **Problems**: The name `_render_series_children` suggests rendering children, but it actually decides whether to spread and then renders, not just pure rendering; `call_` as a protocol name is easily confused with the standard `call` concept.
- **Suggestion**: Can stay as-is. The terms have internal consistency within the project.

### 3.5 Duplicate Logic, Redundant Code & Dead Code

- **Good**: `_grid_cell` and `_table_cell` have similar parameter sets but correspond to Typst's `grid.cell` and `table.cell` respectively, which are semantically different and documented differently in Typst. Keeping two independent definitions is correct.
- **Problem**: In `protocols.py`, `normal` (lines 109-135), `call_` (150-160), `instance` (163-183), `pre_series` (186-201), and `post_series` (204-222) share a common pattern for building `params` lists.
- **Suggestion**: Can stay as-is. The repeated pattern is clear enough, and abstraction could lead to over-coupling. This is a classic case of "keeping clear local repetition."

### 3.6 API Design & Data Contracts

- **Good**: All `std/` functions return `str`, making the API contract consistent; `Document`'s `add_*` methods return `None` with clear semantics; parameter validation provides useful error messages.
- **Problem**: Default values for public parameters use Python literals rather than Typst dialect strings. For example, `page`'s `margin='auto'` rather than `'"auto"'`. This is intentional (the Python-layer `"auto"` is rendered as Typst's `auto`), but it is inconsistent with the quote style of `paper='"a4"'`.
- **Suggestion**: Can stay as-is. This is a convention issue, not an error; maintain internal consistency.

### 3.7 Type Annotations & Static Contracts

- **Good**: Uses modern features like `typing.final`, `ClassVar`, generic `Callable[..., Any]`, `Self`; type hints cover core modules.
- **Problems**:
  1. Heavy use of `Any` (e.g., `Callable[..., Any]`, `dict[str, Any]`), especially protocol functions in `_core` accepting `**kwargs: object` instead of more precise types.
  2. `customizations.py` return type `Callable[..., str]` is imprecise; it should actually express that the returned function signature is `(*args, **kwargs) -> str`.
  3. In `protocols.py`, `kwargs` is typed as `dict[str, object]`, but it is actually `Mapping[str, object]`.
- **Suggestions**:
  1. Most of this can stay as-is. Much of the `Any` usage is due to runtime-determined parameter types (the dynamism of Typst DSL); forcing precision would be misleading.
  2. `dict[str, object]` → `Mapping[str, object]` is a low-cost improvement (P2).

### 3.8 Modern Python Features & Idiomatic Writing

- **Good**: Uses `attrs` library's `@frozen`, `WeakKeyDictionary`/`WeakSet`, `MappingProxyType`, `singledispatch`, `contextlib.redirect_stdout`, `pathlib.Path`; minimum Python 3.11 allows modern syntax.
- **Problem**: Type annotations still use `Optional[X]` instead of `X | None`, `Union[X, Y]` instead of `X | Y`, `List[X]` instead of `list[X]` (in some places), `Tuple[X, ...]` instead of `tuple[X, ...]` (in some places).
- **Suggestion**: Can gradually migrate to 3.10+ union syntax (P2).

### 3.9 Functional Style, State Management & Side Effects

- **Good**: The core rendering pipeline is purely functional (input value → output string); `Implement` registry uses `WeakKeyDictionary` and `WeakSet` to avoid memory leaks.
- **Problem**: `Implement.temporary` is global mutable state (a `WeakSet`), implicitly modified at module import time via the `@temporary` decorator. It is not thread-safe, but concurrency is not a concern in this project.
- **Suggestion**: Can stay as-is. The current use case does not involve concurrency; refactoring cost outweighs benefit.

### 3.10 Return Value Design, Iterator & Generator

- **Good**: All `std` functions return `str`, simple and consistent; `_iter_public_function_paths` uses `Iterable` as the return type, semantically correct.
- **Suggestion**: Current design is reasonable; no modification needed.

### 3.11 Error Handling, Exception Types & Diagnosability

- **Good**: `validate_value` provides a clear list of allowed choices; `raise_unknown_fields` provides explicit field names; `_filter_default_kwargs` relies on this to give precise errors.
- **Problems**:
  1. `scripts/_example_collect.py:87` uses overly broad `except Exception` (P2, see key issue table).
  2. A few functions lack `Raises:` sections in docstrings (P2).
- **Suggestion**: Fix by priority.

### 3.12 Paths, Files, I/O & Platform Compatibility

- **Good**: Uses `pathlib.Path` for unified path handling in scripts; `write_text` explicitly specifies `encoding='utf-8'`.
- **Problem**: `generate_readme.py:14-19` has `sys.path.insert` before import statements, an import-time side effect.
- **Suggestion**: Can stay as-is; this is script code, not library code.

### 3.13 Security

- **Not applicable**: The project does not involve user input processing, network requests, databases, or file uploads. Generated Typst code is ultimately executed in the Typst compiler, with no injection risk.

### 3.14 DataFrame / pandas Code Quality

- **Not applicable**: The project does not use pandas.

### 3.15 Determinism, Randomness & Reproducibility

- **No issues**: The project involves no random numbers or external API calls. Tests and fixture generation are deterministic.

### 3.16 Performance, Memory & Scalability

- **Good**: Uses `WeakKeyDictionary` and `WeakSet` to avoid circular references and memory leaks from the registry; `frozenset` for constant sets with high query efficiency.
- **Suggestion**: Current performance is sufficient; no optimization needed.

### 3.17 Concurrency, Batch Execution & Resource Isolation

- **Not applicable**: The project is a single-threaded code generator with no concurrency or batch processing.

### 3.18 Dependency Management & Engineering Configuration

- **Good**: `pyproject.toml` centralizes configuration; minimal dependencies (only `attrs` and `deprecated`); uses `dependency-groups` for dev dependencies; `uv.lock` locks versions.
- **Problems**:
  1. `ruff.lint.select` rule set is too narrow (P3).
  2. `typstpy.egg-info/` directory appears to be committed to the repository (based on glob results); it should normally be excluded by `.gitignore`.
- **Suggestion**: Gradually expand ruff rules; ensure `egg-info` is in `.gitignore`.

### 3.19 Test Design & Coverage

- **Good**: Tests cover core protocols, each `std` module, `Document`, `customizations`, public API, docs/extraction, Typst fixture generation and compilation; uses `parametrize` for boundary tests; Typst fixture tests actually compile the generated Typst code if the CLI is available.
- **Problems**:
  1. `test_typst_fixture.py:16-30` runs a script via `subprocess.run` and compares output, testing the full pipeline but slow.
  2. Some boundary conditions are lightly tested: various protocol renderings with empty children, deeply nested dict/list rendering.
- **Suggestion**: Current coverage is adequate; selective additions are optional (P2).

### 3.20 Documentation, Docstrings & Comments

- **Good**: Every `std` function has a Typst documentation verification comment and doctest example; Google-style docstrings are uniform; `Raises:` sections are present in most functions.
- **Problems**: Some functions lack `Raises:` (in key issue table); private helper functions in `protocols.py` (`_filter_default_kwargs`, `_should_spread_single_child`, `_render_series_children`) lack docstrings.
- **Suggestion**: Add missing docstrings and `Raises:` sections (P2).

### 3.21 Backward Compatibility & Deprecation Strategy

- **Good**: The `deprecated` library is used to mark `image.decode`, `path`, `pattern`; the `page` function manually handles the `footer_ascent` compatibility alias.
- **Problem**: The removal timeline and version plan for deprecation warnings are not recorded in the code (the `deprecated` decorator only has `version='1.1.1'`, with no indication of when they will be removed).
- **Suggestion**: Can stay as-is. Migration guides can be added when the project approaches the deprecation version.

## 5. Designs That Can Stay As-Is

1. **`Document` class**: A simple list accumulator + `__str__` rendering, with no unnecessary abstraction.
2. **`_docs.py` docstring extraction logic**: The implementation of `_SECTION_HEADING_RE` and `extract_examples` is clear and practical, with reasonable error handling.
3. **Five protocol functions** (`normal`/`instance`/`positional`/`pre_series`/`post_series`): Each has clear semantics; names are internally consistent even if uncommon; repetition among them is “clearer local repetition.”
4. **Independent definitions of `_grid_cell` and `_table_cell`**: They correspond to different Typst documents and different functions; maintaining them independently is correct.
5. **Quoting conventions in `_constants.py`**: Strings with embedded quotes are used because Typst `validate_value` compares values by exact equality — the design is sound.
6. **`sys.path.insert` in `scripts/`**: This is script code, not library code; the import-time side effect is acceptable.
7. **Mild repetition in `protocols.py` params construction**: The five functions each have subtle differences; abstraction would make them harder to understand (see Location 4).
8. **`singledispatch` fallback registration**: Basic types like `int`/`float` are rendered via `RawExpr(str(obj))` — an intentional design choice that is concise and effective.

## 6. Phased Modification Plan

### Summary of the Most Important 3 Changes

1. Fix the fragility of `strip_brace` (P0): the current `value[1:-1]` assumes all values start with `(` and end with `)`. Add a `render_content()` method to IR nodes that returns content without outer parentheses.
2. Fix the spread logic for `#color.map.` (P0): `_SPREADABLE_CODE_PREFIXES` starts with `#` but is stripped during rendering, rendering the current logic ineffective.
3. Add `where`/`with_` to `customizations.py` factory functions (P1): close the factory function feature gap.

### First Batch: Must Fix (P0 + important P1)

- `_core/ir.py`: Add a `render_content()` method to the `TypstExpr` protocol.
- `_core/render.py`: Refactor `strip_brace` to use `render_content()`; fix the `#color.map.` spread prefix matching.
- `_core/protocols.py`: Update `_render_series_children` calls; fix spread logic.
- `customizations.py`: Add `where` and `with_` methods to factory functions.

### Second Batch: Suggested Fixes (general P1 + high-priority P2)

- `scripts/_example_collect.py`: Narrow the `except Exception` scope.
- `_core/protocols.py`: Add docstring to `_should_spread_single_child` or split the function.
- `std/**/*.py`: Add missing `Raises:` docstring sections.
- `_core/render.py`: Fix the `warnings.warn` stacklevel for unregistered functions.

### Third Batch: Gradual Cleanup (remaining P2, documentation, test supplements, etc.)

- Migrate type annotations to 3.10+ union syntax (`X | None` instead of `Optional[X]`).
- Expand the ruff lint rule set.
- Add docstrings for private functions like `_filter_default_kwargs`.
- Change `dict[str, object]` → `Mapping[str, object]` in `_core/protocols.py`.

### Not Recommended for Now

- Large-scale refactoring of protocol functions (repetition is acceptable).
- Merging `call_` and `normal` protocols.
- Migrating the build system (`setuptools` is sufficient).
- Splitting `visualize.py` (1951 lines) — though long, its pattern is uniform; splitting would break existing import relationships.

## 8. Final Conclusion

The overall code quality of the project is very high, especially regarding API design consistency, verification comments, and test coverage. The most critical current issues are the fragility of `strip_brace`'s character operation (P0) and the conflict between the `#color.map.` prefix matching in the spread logic and the rendering pipeline's `#` stripping (P0) — both can lead to incorrect Typst output. The most worth rethinking implementation is the functional completeness of `customizations.py` factory functions (P1), which currently lack `where`/`with_` methods.

The next step should address the P0 items (`strip_brace` + spread logic). Both changes are concentrated in the `_core/` module, with controllable risk and good test coverage. What should not be rushed are local repetitions among std functions (like block/box) and the repetitive params construction in the protocols — those are design choices, not defects, and keeping the current code is clearer.

The project as a whole is fit for continued extension. The existing architectural pattern (`@implement` + IR + `singledispatch`) supports steadily adding new Typst functions, the core engine is stable, and the test suite can catch most regressions. It is recommended to gradually complete the above P1/P2 cleanups while adding new functions.
