# Code Review

## 1. Overall Assessment

- The project is overall healthy, with a clear goal: a lightweight Typst code string generation library, using a standard `src` layout for `src/typstpy` and a test directory that mirrors the source structure.
- The core architecture is reasonable: `_core` handles registration, rendering protocols, and common rule functions; `std/*` splits by Typst standard library domains; `Document` handles document assembly; `scripts/` manages documentation and fixture generation.
- The biggest current risk is not architectural decay, but contract drift among source behavior, README, docstrings, type declarations, and generation scripts.
- The highest-priority issue: the `image` docs claim to support raw bytes, but the underlying renderer treats bytes as `Iterable[int]`, rendering meaningless output; the README has clearly fallen behind the current source and tests.
- Another long-term risk: the project includes `py.typed`, but many public APIs lack return type annotations, and dynamically attached helpers like `.where`, `.with_`, `.caption`, `.cell` lack static contracts.
- The `_core` protocol functions, `implement` decorator, and thin-wrapper pattern can stay as-is; do not refactor into AST, configuration-driven generation, or a complex class hierarchy.
- `std/visualize.py`, `std/model.py`, and `std/layout.py` are large files but their content is highly homogeneous; no need to split by line count right now.
- The testing structure is good, covering core protocols, main public API, fixture generation, and some Typst CLI compilation, but it lacks tests for README drift, bytes contract, type contracts, and script failure paths.
- The project has no pandas/DataFrame, randomness, complex I/O, or obvious performance bottlenecks; related risks are low.
- This review is read-only; `pytest` or `pyright` were not run because the current plan phase would generate coverage/temporary output files.

## 3. Thematic Analysis

### Project Structure

- **Good**: Clear separation among `src/typstpy`, `tests/typstpy`, `scripts`. `std/layout.py`, `std/model.py`, `std/text.py`, `std/visualize.py` split by Typst standard library domains – reasonable boundaries. `_core` further split into `decorators.py`, `registry.py`, `render.py`, `protocols.py`, clearer than a single core file. `tests/typstpy/std/*` mirrors `src/typstpy/std/*` for easy test navigation.
- **Problems**: `std/visualize.py` and `std/model.py` are large but mostly homogeneous wrappers, not mixed responsibilities. `std` modules have cross-imports for docstring examples (e.g., `layout.py` imports `lorem`, `rect`; `model.py` imports `hspace`, `repeat`, `image`, `line`). README generation table exposes internal function names like `_color_map`, `_grid_cell`, `_table_cell`.
- **Modify?**: Do not split `visualize.py` or `model.py` now. Fix document public display names and README drift first. Cross-module example imports can stay for now, but later the example collector should provide a unified namespace.
- **Risk to avoid**: Do not change public import paths just to “make modules smaller”. Do not convert wrappers to config-driven generation tables; readability would drop.

### Abstraction Boundaries

- **Good**: `implement` registers Typst original function names; `normal`/`positional`/`instance`/`pre_series`/`post_series` handle rendering protocol – necessary abstractions. `std` wrappers remain thin functions for easy Typst-doc mapping. `_docs.py` uses dataclass for `ImplementRecord` and `ExampleBlock`, clear responsibilities.
- **Problems**: `implement` dynamically attaches `.where` and `.with_`; `attach_func` dynamically attaches sub-functions – good at runtime but static contract unclear. `_render_value`’s `Iterable` branch too broad, causing the `bytes` contract issue. `scripts/_example_collect.py` combines parsing doctest, executing source, filtering Typst results, and recording skips; responsibilities slightly concentrated.
- **Modify?**: Keep `_core` protocol abstractions; do not replace with class hierarchy. Add type layer or stubs for dynamic public helpers instead of rewriting the decorator mechanism. The example collector can be slightly split, but no new framework needed.
- **Risk to avoid**: Do not change the return type of all wrappers. Do not over-model Typst expressions into complex objects.

### Implementation Quality

- **Good**: Core generation logic is mostly pure string transformation, easy to test. `_validate_value` error messages include function label, field, and allowed values – good diagnosability. `iter_registered_functions()` is sorted, so README/fixture generation order is deterministic.
- **Problems**: Boundaries among bytes, PathLike, and Typst string literals are not explicitly expressed. README and source behavior have drifted, indicating generated docs are not a quality gate. Multiple mutable defaults do not follow modern Python public API practices. `scripts/_example_collect.py` is too forgiving of execution failures.
- **Modify?**: Prioritize fixing contracts and tests; no large-scale refactoring. `_render_value` design can stay, but needs special-type branching or rejection policy.
- **Risk to avoid**: Do not automatically add Typst quotes to all Python strings – that would break existing API habits. Do not implement complex serialization for arbitrary Python objects just to support `bytes`.

### Naming Consistency

- **Good**: `set_`, `show_`, `import_` sensibly avoid Python keywords. `bullet_list` for Typst `list`, `numbered_list` for Typst `enum` – reasonable naming that avoids built-in collision. `hspace`, `vspace` are more Pythonic than `h`, `v`. `original_name` clearly expresses the original Typst name.
- **Problems**: README table “Package's function name” shows private implementation names for attached helpers, not the user’s actual entry point. `instance(func, instance, /, ...)` – parameter name `instance` repeats function name, internal readability mediocre. `self`, `instance`, `target` mixed in method-like APIs. “typst”/“Typst” capitalization inconsistent.
- **Terms to unify**:
  - **Typst code**: Capitalize as “Typst” uniformly.
  - **original_name**: keep for Typst original function name.
  - **children**: keep for series parameters.
  - **target** or **self**: unify inside method-like wrappers; public docstrings can keep `self` meaning the Typst receiver.
  - **attached helper**: for dynamically attached functions like `.caption`, `.cell`, `.map`, `.decode`.
- **Recommended to keep**:
  - `set_`, `show_`, `import_`.
  - `bullet_list`, `numbered_list`.
  - `hspace`, `vspace`.
  - `Document`.
  - `_core` internal module split.
- **Suggested to change or clarify**:
  - README display names for attached helpers: from `std.visualize._color_map` to `std.visualize.color.map` or a consistent public path.
  - Internal `instance` parameter can later become `target`, should not affect public API.
  - `numbering_` can stay; not worth renaming a public parameter for aesthetics.
- **Impact**: README display name change only affects docs, not code. Public parameter rename would break keyword calls – not recommended. Static typing of attached helpers may need `.pyi` or protocols, but should not change runtime names.

### Duplicate Logic

- **Good**: The repetition in `std` wrappers is domain API mapping, not bad duplication. `normal`, `post_series`, `pre_series` already extract the truly high-value repetition. `grid` and `table` are similar but in different Typst domains; keeping explicit makes them easier to match against docs.
- **Problems**: `image` and `image.decode` share repeated `format`/`fit`/`scaling` validation. Color constructor “1 or 3/4 arguments” logic repeats. `relative != 'auto'` validation repeats across `gradient`/`pattern`/`tiling`. Mutable default forms appear repeatedly.
- **Modify?**: Can locally extract very small helpers, e.g., `_validate_image_options` or `_validate_relative`. Do not abstract all wrapper parameter schemas yet. Local repetition is clearer, especially for each Typst function's parameter list.
- **Risk to avoid**: Do not replace function definitions with a huge registry/config table. Do not force readers to jump through multiple abstraction layers to understand one wrapper.

### API & Data Contract

- **Good**: Most public wrappers return `str`, simple contract. `Document` is explicitly a mutable builder, `add_*` returns `None`, tests cover it. `__all__` controls top-level and `std` exports. `customizations` provide an escape hatch for unimplemented Typst functions, reasonable.
- **Problems**: `bytes` contract unclear – the most explicit data contract issue. Dynamically attached helpers are real public API but not expressed at the type level. README support table uses internal registry names as public function names. There is no `dict[str, Any]` proliferation, but Typst parameter values are wide-open types; no unified type alias yet.
- **Which interfaces are public-like and need caution**: `typstpy.Document`, all `__all__` exports from `typstpy.std.*`, `typstpy.subpar.grid`, `typstpy.customizations.normal`/`instance`/`positional`/`pre_series`/`post_series`, and dynamic helpers: `.where`, `.with_`, `.caption`, `.cell`, `.line`, `.map`, `.decode`, etc.
- **Which are internal and can be refactored freely**: `typstpy._core.*` private helpers, `typstpy._docs.*` (not top-level public, but scripts/tests depend; changes need care), `scripts/_example_collect.py` internals.
- **Return types not to change lightly**: all wrappers return `str`, `Document.__str__()` returns `str`, `_docs.collect_*` return lists – because generation scripts need stable snapshots.
- **Data structures to make explicit**: `TypstCode = str` or similar type alias; wide Typst value input type can use `object` or a conservative alias, not a complex union yet. Attached helper typing can be expressed with `.pyi` or `Protocol` in stages.

### Type Annotations

- **Good**: Project targets Python >=3.11, modern type syntax available. `_docs.py` uses dataclass and `Callable[..., Any]`. `visualize.rgb` uses `@overload` – direction is correct. Pyright config exists.
- **Problems**: Many public wrappers lack `-> str`. `@overload` missing explicit return types. Dynamic function attributes hard for type checkers to perceive. `py.typed` present but type contract coverage insufficient. `_render_value`’s `Iterable` registration too broad; static layer doesn't express `bytes` special case.
- **Modify?**: Phase 1: only add return types and core helper types, do not over-narrow parameters. Phase 2: add attached helper stubs or `Protocol`. Phase 3: consider `Literal` for a few stable value domains like `fit`, `scaling`, `fill_rule`, but do not cover all Typst string expressions.
- **Risk to avoid**: Do not forbid users from passing Typst native expression strings for type safety. Do not replace Typst open strings with lots of `Enum`/`StrEnum`. Do not write giant unions for all parameters at once – maintenance cost would be high.

### Modern Python Features

- **Good**: Uses `list[str]`, `X | None`, `dataclass(frozen=True)`, `pathlib.Path`, `collections.abc`. `singledispatch` for rendering different Python values – clean and effective. Positional-only and keyword-only parameters used well for wrapper semantics. `scripts` uses `argparse` and `Path`, no hard-coded Windows paths.
- **Problems**: Mutable defaults inconsistent with modern Python public API style. Could use `slots=True` on small dataclasses but benefit limited. Could use type alias to simplify Typst value contract. Not suitable for mechanically introducing pattern matching, generators, or complex `Protocol` everywhere.
- **Recommended modern constructs**: `-> str`, `-> None` return types. `TypeAlias` for `TypstCode`. `Mapping`/`Sequence` for read-only input contracts only where truly helpful. `Path` kept at scripts layer; no rush to support `PathLike` in core.
- **Not recommended**: Comprehensive `Enum`-ification of Typst string values. Converting all wrappers to dataclasses or class methods. Changing series output to generators. Adding new dependencies for runtime schema validation.
- **Python version constraint**: `requires-python = ">=3.11"` allows modern type syntax. Avoid Python 3.12 `type Alias = ...` syntax to keep 3.11 compatibility. Adding return types does not break; changing parameters to keyword-only or positional-only can break callers – caution needed. Changing `bytes` from silent wrong rendering to `TypeError` is a behavior change but a clear contract fix.

### Functional Style & Side Effects

- **Good**: Most wrappers are pure functions: input parameters, output Typst strings. `_render_value`, `normal`, `post_series` etc. easy to unit-test. `Document` side effects are confined to the builder instance; tests verify `add_*` return `None`.
- **Problems**: `implement` and `attach_func` modify global registries and function attributes at import/decorator time – necessary but implicit side effects. `scripts/_example_collect.py` executes docstring examples, which has execution-side-effect risk. README/fixture generation depends on import registration order, though sorting reduces instability.
- **Should become pure**: README rendering functions are already near-pure; keep as-is. Fixture rendering can stay: input example, output string.
- **Side effects allowed**: decorator registration, `Document.add_*`, `generate_typst_fixture.py --write` writing files.
- **Side effects that must be documented**: `implement` registration and dynamic `.where`/`.with_` attachment; `attach_func` dynamic helper attachment; example collector executing doctest.
- **Do not force functional style**: do not make `Document` immutable builder; do not explicitly pass registry state to every wrapper.

### Return Value Design & Generators

- **Is current design reasonable?**: Yes. Project goal is generating Typst source; `str` return fits consumers best. `collect_example_blocks()` and `collect_implement_records()` return lists – callers need sorting, stable snapshots, repeated iteration. `Document.__str__()` returning full document string is reasonable.
- **What could be a generator?**: Currently no public API obviously suited for generators. If README generation content becomes very large, internal streaming could be used, but not needed now.
- **What should not be a generator?**: All Typst wrappers, `Document.__str__()`, README/fixture generation public script helpers (comparing full strings is more intuitive for tests).
- **Would changing to generator break API/tests?**: Yes, would break almost all tests and user call patterns; not recommended.
- **Where list/tuple/str is more appropriate**: `_docs.collect_*` keep as list, wrappers keep as str, fixed metadata keep as dataclass.

### Error Handling

- **Good**: Unknown fields use `TypeError`, illegal values use `ValueError` – direction is reasonable. `_validate_value` error message includes function label, field, and values. Unregistered `Callable` uses `warnings.warn(..., stacklevel=2)` to locate caller.
- **Problems**: Some docstring `Raises` do not match actual exceptions. Example collector’s `except Exception` swallows execution failures. `warnings.warn` does not specify warning category; default `UserWarning` is acceptable but could be explicit. `generate_typst_fixture.py` does not fail by default on skipped execution errors.
- **Must modify**: `bytes` contract corresponding error or implementation; example collector failure exit strategy.
- **Can keep**: `_raise_unknown_fields` using `TypeError`; `_validate_value` using `ValueError`; `page` compatibility alias conflict using `TypeError`.
- **Error messages needing context**: collector skip summary should include qualname, example index, source, and exception. Bytes rejection should explain that the Python wrapper currently accepts Typst path/string literal, not raw Python bytes, or how to use it.
- **Custom exception types needed?**: Not now. `TypeError`, `ValueError`, `RuntimeError` are sufficient.

### Paths & I/O

- **Good**: Core library basically no file I/O. Scripts use `Path`, cross-platform risk low. README generation defaults to stdout, clipboard needs explicit `--clipboard`. Fixture generation only writes with `--write`.
- **Problems**: `scripts/generate_readme.py` and `_example_collect.py` both manually insert `SRC_ROOT` into `sys.path` – slight repetition. `image(source)` docs mention path or bytes, but Python `Path` contract unclear, and whether strings are auto-quoted is not explained. pytest config writes coverage XML; test commands are not strictly read-only.
- **Modify?**: Do not recommend core layer auto-handle `Path` and add quotes, as existing API expects users to pass Typst literal string. Clarify in docstring that `source` should be a Typst source fragment like `'"image.png"'`. Scripts’ `sys.path` handling can stay unless scripts multiply.
- **Risk to avoid**: Do not automatically render `image(Path("x.png"))` to `x.png` or `"x.png"` without explicit design and testing. Do not let generation scripts write files or access clipboard on import.

### DataFrame Schema

- **Good**: Project does not use pandas/DataFrame.
- **Problems**: None.
- **Modify?**: Not applicable.
- **Risk to avoid**: Do not introduce pandas for table docs; standard library rendering is sufficient.

### Determinism & Reproducibility

- **Good**: `iter_registered_functions()` sorted by `(__module__, __name__)`. `_validate_value` sorts allowed choices for output. Python 3.11 dict insertion order is stable; kwargs rendering order reproducible. Fixture test compares generated output against committed fixture.
- **Problems**: README lacks equivalent drift test. Dynamic `customizations` registrations depend on object lifetime and global registry, not suitable for concurrent/long-term cache scenarios. Example collector executes source; exception skipping policy affects fixture reproducibility.
- **What outputs might be unstable**: If README is not committed after script generation, docs can drift. If doctest examples depend on external environment or warning/print, collector behavior may vary.
- **Where explicit sorting needed**: Registered functions already sorted. If future tables are sorted by attached public path, sort that path too.
- **Random processes needing seed**: Project has no random processes.
- **What config to record in results**: Fixture header already records generation script. README generation doesn't need extra metadata but should have test protection.

### Performance

- **Good**: String generation volume is small, no obvious bottlenecks. `StringIO` for multi-segment assembly is sufficient. No DataFrame concat, deep copy, solver, or graph algorithms.
- **Problems**: `singledispatch` and recursive rendering overhead under extremely deep nesting, but not a problem at current scale. If supporting raw bytes and directly embedding in Typst source, huge strings could be generated – caution needed.
- **Modify?**: Do not rewrite core for performance. If implementing `bytes` support, consider size, encoding, and error messages.
- **Risk to avoid**: Do not sacrifice `_render_value` readability for micro-optimizations. Do not introduce caching; no repeated expensive computations exist.

### Concurrency & Batch Execution

- **Good**: Wrapper functions mostly stateless, low concurrent-call risk. `Document` state encapsulated in instances.
- **Problems**: `_Implement.permanent` and `_Implement.temporary` are global mutable registries. Dynamic `customizations` register temporary functions at runtime. `Document` is not a thread-safe builder.
- **Modify?**: Do not introduce locks or isolated registries now. If multi-threaded generation of custom function docs becomes necessary in the future, then consider registry isolation.
- **Risk to avoid**: Do not rewrite decorator/registry for theoretical thread safety. For batch tasks that run scripts, let each task run in an independent process.

### Dependencies & Engineering Configuration

- **Good**: `pyproject.toml` centralizes build system, project metadata, runtime dependency, dev dependencies, pyright, ruff, pytest. Runtime dependencies minimal – only `deprecated`. `pyperclip` is dev-only and imported only for `--clipboard`. `py.typed` included as package data.
- **Problems**: Ruff rules are few, mostly basic syntax and imports. Pyright is basic and dynamic helper typing issues unresolved. No CI or documented local check command. `[dependency-groups]` is uv-friendly but may need clarification for other tools.
- **Suggested new/adjusted config**: Add minimal check commands to README or development docs: after activating venv, run `python -m pytest`, `pyright`, `ruff check`, `ruff format --check`. CI can be added later, but not the top code-quality priority now. Pyright can stay basic; increase strictness after annotations are fuller.
- **Do not introduce**: mypy alongside pyright unless team explicitly needs it. Pandas, pydantic, or runtime schema dependency. Complex docs framework just for README tables.
- **Minimal executable check commands**:
  - `& ./activate_venv.ps1`
  - `python -m pytest`
  - `pyright`
  - `ruff check .`
  - `ruff format --check .`

### Testing

- **Good**: `test_core.py` covers core protocols and `.where`/`.with_`. `test_public_api.py` covers top-level and std public helpers. `std/test_*.py` cover typical outputs and illegal parameters. `test_typst_fixture.py` covers fixture freshness and compiles when Typst CLI is available. Source doctest participates in pytest.
- **Must add tests**:
  - `image` bytes contract: if supported, test correct Typst output; if rejected, test `TypeError`.
  - README drift test: verify generated README support sections match committed README’s corresponding regions, or at least test README examples no longer contradict source.
  - Example collector unexpected error test: bad doctest should not be silently skipped by default.
  - Dynamic helper typing may not be testable by pytest alone, but needs pyright or stub test protection.
- **Suggested tests**:
  - `subpar.grid` doc example and external package skip behavior.
  - `render_implement_table` test for attached helper public display name.
  - Regression test after mutable default cleanup ensuring empty dict still omitted.
  - `image`/`image.decode` shared `format`/`fit`/`scaling` validation behavior.
- **Tests not needed now**:
  - Unit test for every default parameter of every std wrapper.
  - Test `_render_value` for every Python collection combination.
  - Performance benchmarks.
- **Internal details not to test**: Do not directly depend on `_Implement`'s internal weak dictionary type. Do not test decorator internal closure structure. Do not test exact full-row order of README table unless as a generated-output drift test.
- **Target behaviors per test**:
  - Bytes test: fix public `image` input contract.
  - README drift test: prevent published docs from lagging.
  - Collector failure test: prevent hidden broken examples.
  - Public display name test: prevent internal helpers from leaking into user docs.

### Documentation & Docstrings

- **Good**: Most public wrappers have Google-style docstrings and examples. `_docs.extract_examples()` has tests and can identify Examples sections. README has broad coverage. Better Notes-style Typst docs verification comments starting to appear.
- **Problems**: README outdated. `subpar.py` has `_description_` placeholder. `model.py:783` has Chinese implementation comment. Some `Raises` do not match actual exception types. README support table shows private helper names. Some docstrings mechanically repeat defaults, slightly verbose.
- **Functions/classes needing docstrings**: `subpar.grid` parameter descriptions. `_core` decorator side effect descriptions can be more explicit. `image` source contract.
- **Docstrings that could be shortened**: Many std wrappers' default value descriptions can be gradually streamlined, not priority. Do not mechanically rewrite all docstrings at once.
- **Comments to delete or update**: Chinese implementation comment in `src/typstpy/std/model.py:783`. Stale examples and stale links in README.
- **Design docs to sync**: No `*_design.md` or `*_impl.md` found. README is the most important user-facing doc to sync.
- **Implementation details that should not go into design docs**: `_Implement` using `WeakKeyDictionary`, `_render_value` specific `singledispatch` registration order, individual wrapper every default parameter.

### Backward Compatibility

- **Good**: Deprecated APIs kept via `deprecated`, e.g., `path`, `pattern`, `image.decode`. `page` retains `footer_ascent` compatibility alias. Top-level `__all__` is concise and stable. Wrappers uniformly return `str`, strong compatibility.
- **What can be done directly**: README regenerate, docstring corrections, adding return type annotations, adding tests, correcting public display name in documentation.
- **What needs phased migration**: `image` bytes contract – if changing from silent rendering to `TypeError`, needs changelog. Dynamic helper typing – if introducing `.pyi`, should be gradual, not full modeling at once. Mutable default cleanup can be done module-by-module in batches to avoid huge diffs.
- **What must be cautious**: Renaming functions, parameters, or return types; changing `str` to AST/Code object; changing Typst string literal handling rules; removing deprecated APIs.
- **Keep old interfaces?**: Keep `path`, `pattern`, `image.decode` deprecated aliases. Keep `footer_ascent` alias unless formally deprecated. Keep all current `std.__all__` public names.

## 4. Evaluation of Alternative Implementation Approaches

### Alternative A: Conservative Local Improvements

- **Core idea**: Do not change architecture; fix contract drift, README drift, bytes contract, script failure strategy, and low-risk type annotations first.
- **Files involved**: `src/typstpy/std/visualize.py`, `src/typstpy/_core/render.py`, `src/typstpy/_docs.py`, `scripts/_example_collect.py`, `scripts/generate_readme.py`, `README.md`, `tests/typstpy/*`.
- **Change scope**: Small to medium, mostly local conditional branches, doc generation, test additions.
- **Pros**: Lowest risk, best fits “review first, no large refactoring” goal.
- **Cons**: Will not solve all type and dynamic helper contract issues at once.
- **Does it change public API?**: Possibly `bytes` input behavior; otherwise mostly not.
- **Need new tests?**: Yes.
- **Applicability**: Best for current phase.

### Alternative B: Medium-scale Refactoring

- **Core idea**: On top of A, systematically complete public return types, add `.pyi` or `Protocol` for dynamic helpers, clean mutable defaults, improve collector state model.
- **Files involved**: `src/typstpy/_core/*`, `src/typstpy/std/*.py`, `src/typstpy/customizations.py`, `scripts/*.py`, `tests/*`, `pyproject.toml`.
- **Change scope**: Medium, across modules but does not change core architecture.
- **Pros**: Significantly boosts typed package trustworthiness and long-term maintainability.
- **Cons**: Larger diff, easy to mix with behavior fixes.
- **Does it change public API?**: Theoretically not function names or return values, but may change exception types or `bytes` behavior.
- **Need new tests?**: Yes, need both pytest and pyright verification.
- **Applicability**: Second phase after A is done.

### Alternative C: More Ideal but Riskier Design

- **Core idea**: Model Typst source fragments as lightweight Code/AST objects, then unified render; solve escaping, bytes, typing, composition completely.
- **Files involved**: Almost all source, tests, README.
- **Change scope**: Large.
- **Pros**: Best long-term expressiveness.
- **Cons**: Breaks `str` return habit; requires rewriting tests and docs; not worth at current project scale.
- **Does it change public API?**: High probability.
- **Need new tests?**: Need to rebuild most tests.
- **Applicability**: Not recommended; consider only when the project clearly needs complex Typst AST operations.

### Recommended Approach

- **Recommend Alternative A**. The project’s main issues are contract and documentation drift, not core architecture flaws. First use local, verifiable fixes to align public API, README, fixture, and type promises, then enter B for gradual types and default cleanup.

## 5. Designs That Can Stay As-Is

- `_core` rendering protocol abstraction can stay: `normal`, `positional`, `instance`, `pre_series`, `post_series` are necessary reuse points.
- `std` wrapper explicit function definitions can stay: although repetitive, they are readable and easy to match against Typst docs.
- All wrappers returning `str` can stay: the most stable and testable public contract.
- `Document` as a mutable builder can stay; no need to change to immutable API.
- `std/layout.py`, `std/model.py`, `std/text.py`, `std/visualize.py` domain split can stay; no need to split just because files are large.
- Pythonic names like `bullet_list`, `numbered_list`, `hspace`, `vspace` can stay.
- Deprecated aliases can stay; do not delete.
- Local wrapper repetition can stay; do not abstract into a large config table.
- No need for generators; current string snapshots suit testing and debugging better.
- No need for pandas/DataFrame schemas.
- No need for a custom exception hierarchy; standard exceptions suffice.
- No need to optimize `_render_value` for theoretical performance issues.
- Do not change to Typst AST/Code object for now.

## 9. Final Conclusion

The most critical issue in the current project is not architectural design, but contract synchronization: a closed loop among source, README, docstrings, fixture generation, and type declarations hasn't been fully achieved. The most worth rethinking are the `image` bytes/path/source representation and the public API expression of dynamic attached helpers in documentation and the type system.  
The next steps should first fix README drift and `image` bytes contract, as they are already user-visible problems. Then proceed with type annotations, attached helper display names, example collector failure strategy, and mutable default cleanup.  
What should not be rushed: the `std` wrapper architecture, return value types, module splitting, AST-ification, and config-table generation. The current thin-wrapper + `_core` protocol layer is suitable; the project is fit for continued extension, but before extending, the documentation generation and type contract quality lines must be solidified.
